üìö Introduction: The Power to Modify Data

In the previous chapters, you learned how to retrieve data using SELECT statements. Now it's time to learn how to modify that data. This chapter covers the three fundamental DML operations that, along with SELECT, form the complete CRUD operations (Create, Read, Update, Delete) that every database developer uses daily.

üìù Real-World Analogy: The Digital Notebook

Think of a database table like a digital notebook:

  • INSERT is like adding new pages with content
  • UPDATE is like erasing and rewriting parts of existing pages
  • DELETE is like tearing out entire pages
  • SELECT (from previous chapters) is like reading the pages

Just like with a real notebook, you need to be careful - once you modify or remove something, the original is gone!

üîß How to Create Test Tables

Before modifying real data, it's crucial to practice on test tables. This section shows you how to create safe environments for learning.

Creating Tables for This Book

First, let's create a simple test database and tables to practice with:

-- Create a test database
CREATE DATABASE IF NOT EXISTS test_db;
USE test_db;

-- Create a sample customers table
CREATE TABLE customers (
    customer_id INT PRIMARY KEY AUTO_INCREMENT,
    first_name VARCHAR(50) NOT NULL,
    last_name VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE,
    phone VARCHAR(20),
    registration_date DATE DEFAULT (CURRENT_DATE),
    credit_limit DECIMAL(10,2) DEFAULT 1000.00
);

Creating a Copy of a Table

One of the safest ways to practice DML statements is to create copies of existing tables. Here are three methods:

-- Method 1: Copy only the structure (no data)
CREATE TABLE customers_backup 
LIKE customers;

-- This creates an exact copy of the table structure
-- including indexes, but without any rows
-- Method 2: Copy structure and all data
CREATE TABLE customers_copy AS
SELECT * FROM customers;

-- Note: This doesn't copy indexes or auto_increment properties
-- You'll need to add those manually if needed
-- Method 3: Copy with selective data
CREATE TABLE vip_customers AS
SELECT 
    customer_id,
    first_name,
    last_name,
    email,
    credit_limit
FROM customers
WHERE credit_limit > 5000;
Always work on copies of tables when learning or testing. Use names like _test, _backup, or _copy to clearly identify them.

Verifying Your Test Tables

-- Check if your tables were created successfully
SHOW TABLES;

-- Verify the structure
DESCRIBE customers_backup;

-- Check if data was copied (if applicable)
SELECT COUNT(*) FROM customers_copy;

‚ûï How to Insert New Rows

The INSERT statement adds new rows to a table. It's like adding new records to your database filing system.

Inserting a Single Row

The most basic INSERT adds one row at a time:

-- Method 1: Specify all columns and values
INSERT INTO customers 
(first_name, last_name, email, phone, registration_date, credit_limit)
VALUES 
('John', 'Doe', 'john.doe@email.com', '555-0101', '2024-01-15', 2500.00);

-- Method 2: If providing values for ALL columns in order
INSERT INTO customers
VALUES 
(NULL, 'Jane', 'Smith', 'jane.smith@email.com', '555-0102', '2024-01-15', 3000.00);
-- NULL for auto-increment customer_id

üéØ Analogy: The Form Submission

INSERT is like filling out a form: you specify which fields you're filling (column names) and what you're writing in each field (values). If a field has a default value, it's like a pre-filled form field - you can leave it blank and it uses the default.

Inserting Multiple Rows

You can insert multiple rows in a single statement for better performance:

-- Insert multiple customers at once
INSERT INTO customers 
(first_name, last_name, email, phone)
VALUES 
    ('Alice', 'Johnson', 'alice.j@email.com', '555-0201'),
    ('Bob', 'Williams', 'bob.w@email.com', '555-0202'),
    ('Carol', 'Brown', 'carol.b@email.com', '555-0203'),
    ('David', 'Davis', 'david.d@email.com', NULL);
-- Notice: We omitted registration_date and credit_limit (will use defaults)
Inserting multiple rows in one statement is much faster than multiple single-row inserts, especially for large datasets. MySQL only needs to parse the query once and can optimize the insertion.

Viewing Your Insertions

-- Check what was inserted
SELECT * FROM customers 
ORDER BY customer_id DESC 
LIMIT 5;

-- Verify the auto-generated IDs
SELECT LAST_INSERT_ID();  -- Returns the last auto-generated ID

üöÄ Advanced INSERT Techniques

Working with Default Values and NULLs

Understanding how to use defaults and NULL values is crucial for efficient data insertion:

-- Using DEFAULT keyword explicitly
INSERT INTO customers 
(first_name, last_name, email, registration_date, credit_limit)
VALUES 
('Emily', 'Wilson', 'emily.w@email.com', DEFAULT, DEFAULT);

-- Mixing NULL and DEFAULT values
INSERT INTO customers 
(first_name, last_name, email, phone, credit_limit)
VALUES 
('Frank', 'Miller', 'frank.m@email.com', NULL, DEFAULT);
-- NULL means "no value", DEFAULT uses the column's default value

NULL vs DEFAULT vs Empty String

  • NULL: Represents the absence of any value (unknown or not applicable)
  • DEFAULT: Uses the column's predefined default value
  • Empty String (''): An actual value - a string with zero characters
  • Omitting the column: Also uses the default value if one exists

Using Subqueries in INSERT Statements

You can use SELECT statements to provide values for INSERT, allowing you to copy and transform data:

-- Insert data from another table
INSERT INTO vip_customers 
(first_name, last_name, email, credit_limit)
SELECT 
    first_name, 
    last_name, 
    email,
    credit_limit * 1.5 AS new_limit
FROM customers
WHERE credit_limit > 3000
    AND registration_date < '2024-01-01';

-- Insert summary data into a stats table
INSERT INTO monthly_stats 
(month, year, new_customers, avg_credit)
SELECT 
    MONTH(registration_date) AS month,
    YEAR(registration_date) AS year,
    COUNT(*) AS new_customers,
    AVG(credit_limit) AS avg_credit
FROM customers
GROUP BY YEAR(registration_date), MONTH(registration_date);

INSERT with Duplicate Key Handling

MySQL provides ways to handle conflicts when inserting data that might violate unique constraints:

-- Option 1: INSERT IGNORE - Skip if duplicate
INSERT IGNORE INTO customers 
(first_name, last_name, email)
VALUES 
('John', 'Doe', 'john.doe@email.com');
-- If email already exists, this row is skipped silently

-- Option 2: ON DUPLICATE KEY UPDATE - Update if exists
INSERT INTO customers 
(first_name, last_name, email, credit_limit)
VALUES 
('John', 'Doe', 'john.doe@email.com', 5000)
ON DUPLICATE KEY UPDATE
    credit_limit = credit_limit + 500,
    last_name = VALUES(last_name);
-- If email exists, increase credit limit and update last name

-- Option 3: REPLACE - Delete old and insert new
REPLACE INTO customers 
(customer_id, first_name, last_name, email)
VALUES 
(1, 'John', 'Doe Updated', 'john.new@email.com');
-- Deletes the old row with customer_id=1 and inserts this new one
REPLACE is destructive - it actually deletes the old row and inserts a new one. This can trigger DELETE and INSERT triggers and reset auto-increment values. Use with caution!

‚úèÔ∏è How to Update Existing Rows

The UPDATE statement modifies existing data in your tables. It's one of the most powerful yet potentially dangerous operations if not used carefully.

‚úèÔ∏è Analogy: The Editor's Pen

UPDATE is like being an editor with a red pen. You can change specific words (columns) in specific sentences (rows) of your document (table). But unlike a word processor, there's no built-in undo button - once you save (commit), the changes are permanent!

Basic UPDATE Syntax

-- Update a single column for specific rows
UPDATE customers
SET credit_limit = 5000
WHERE customer_id = 1;

-- Update multiple columns
UPDATE customers
SET 
    credit_limit = credit_limit * 1.1,  -- 10% increase
    phone = '555-9999'
WHERE last_name = 'Doe'
    AND registration_date < '2024-01-01';
ALWAYS use a WHERE clause unless you intend to update ALL rows! Forgetting WHERE updates every row in your table!

Safe Update Practices

The SELECT-First Rule

Before running any UPDATE, first run a SELECT with the same WHERE clause to see what will be affected:

-- Step 1: Check what will be updated
SELECT * FROM customers
WHERE credit_limit < 2000
    AND registration_date > '2023-01-01';

-- Step 2: If results look correct, run the update
UPDATE customers
SET credit_limit = credit_limit * 1.25
WHERE credit_limit < 2000
    AND registration_date > '2023-01-01';

UPDATE with Calculations and Functions

-- Using calculations in SET clause
UPDATE customers
SET 
    credit_limit = CASE
        WHEN credit_limit < 1000 THEN credit_limit * 1.5
        WHEN credit_limit < 5000 THEN credit_limit * 1.2
        ELSE credit_limit * 1.1
    END;

-- Using string functions
UPDATE customers
SET 
    email = LOWER(email),
    last_name = UPPER(SUBSTRING(last_name, 1, 1)) 
                  + LOWER(SUBSTRING(last_name, 2))
WHERE customer_id > 100;

UPDATE with Subqueries

You can use subqueries to update based on data from other tables or complex calculations:

-- Update based on data from another table
UPDATE customers c
SET credit_limit = (
    SELECT AVG(total_amount) * 10
    FROM orders o
    WHERE o.customer_id = c.customer_id
        AND o.order_date > DATE_SUB(NOW(), INTERVAL 6 MONTH)
)
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.customer_id
);

-- Update using JOIN syntax (MySQL specific)
UPDATE customers c
INNER JOIN (
    SELECT customer_id, COUNT(*) AS order_count
    FROM orders
    GROUP BY customer_id
) order_stats
ON c.customer_id = order_stats.customer_id
SET c.credit_limit = c.credit_limit + (order_stats.order_count * 100);

Tracking UPDATE Results

-- Check how many rows were affected
UPDATE customers
SET phone = 'NOT PROVIDED'
WHERE phone IS NULL;

-- MySQL will return: "X rows affected"
-- You can also use ROW_COUNT() function
SELECT ROW_COUNT() AS rows_updated;

üóëÔ∏è How to Delete Existing Rows

The DELETE statement removes rows from a table. It's the most destructive DML operation and should be used with extreme caution.

üóëÔ∏è Analogy: The Shredder

DELETE is like putting documents through a shredder. Once they're gone, they're gone forever (unless you have backups). Unlike moving files to a recycle bin, there's no easy way to recover deleted database rows. Always double-check what you're about to shred!

Basic DELETE Syntax

-- Delete specific rows
DELETE FROM customers
WHERE customer_id = 5;

-- Delete multiple rows based on condition
DELETE FROM customers
WHERE registration_date < '2020-01-01'
    AND credit_limit = 0;

-- Delete with LIMIT (useful for batch processing)
DELETE FROM customers
WHERE email IS NULL
ORDER BY customer_id
LIMIT 10;
DELETE without WHERE removes ALL rows! The table structure remains, but all data is gone. Use TRUNCATE TABLE for better performance when deleting all rows intentionally.

Safe DELETE Practices

-- Always SELECT first to verify what will be deleted
-- Step 1: Check what will be deleted
SELECT * FROM customers
WHERE last_login < DATE_SUB(NOW(), INTERVAL 2 YEAR);

-- Step 2: Count the rows
SELECT COUNT(*) FROM customers
WHERE last_login < DATE_SUB(NOW(), INTERVAL 2 YEAR);

-- Step 3: If correct, delete
DELETE FROM customers
WHERE last_login < DATE_SUB(NOW(), INTERVAL 2 YEAR);
-- Create a backup before deleting
CREATE TABLE customers_deleted_backup AS
SELECT *, NOW() AS deleted_at
FROM customers
WHERE status = 'inactive';

-- Now safe to delete
DELETE FROM customers
WHERE status = 'inactive';
-- Instead of DELETE, use a "soft delete" approach
-- Add a deleted flag column to your table
ALTER TABLE customers
ADD COLUMN is_deleted BOOLEAN DEFAULT FALSE,
ADD COLUMN deleted_at DATETIME DEFAULT NULL;

-- "Delete" by marking as deleted
UPDATE customers
SET 
    is_deleted = TRUE,
    deleted_at = NOW()
WHERE customer_id = 10;

-- Filter out "deleted" records in queries
SELECT * FROM customers
WHERE is_deleted = FALSE;

DELETE with Subqueries

You can use subqueries to delete based on complex conditions:

-- Delete customers with no orders
DELETE FROM customers
WHERE customer_id NOT IN (
    SELECT DISTINCT customer_id 
    FROM orders
);

-- Alternative using NOT EXISTS (often faster)
DELETE FROM customers c
WHERE NOT EXISTS (
    SELECT 1 
    FROM orders o
    WHERE o.customer_id = c.customer_id
);

-- Delete using JOIN (MySQL specific)
DELETE c 
FROM customers c
LEFT JOIN orders o 
    ON c.customer_id = o.customer_id
WHERE o.customer_id IS NULL;

TRUNCATE vs DELETE

When to use each:

Feature DELETE TRUNCATE
Can use WHERE clause ‚úì Yes ‚úó No
Speed for all rows Slower Much faster
Triggers fired Yes No
Can rollback Yes (in transaction) No
Resets AUTO_INCREMENT No Yes 
-- Remove all rows with DELETE (slow, logged)
DELETE FROM large_table;

-- Remove all rows with TRUNCATE (fast, not logged)
TRUNCATE TABLE large_table;

‚ö° Best Practices and Safety Tips

Transaction Safety

Use transactions to group related changes and enable rollback if something goes wrong:

-- Start a transaction
START TRANSACTION;

-- Perform your operations
UPDATE accounts 
SET balance = balance - 100
WHERE account_id = 1;

UPDATE accounts 
SET balance = balance + 100
WHERE account_id = 2;

-- Check if everything looks good
SELECT * FROM accounts 
WHERE account_id IN (1, 2);

-- If good, commit; if not, rollback
COMMIT;    -- Makes changes permanent
-- OR
ROLLBACK;  -- Undoes all changes since START TRANSACTION

Performance Optimization

Tips for Faster DML Operations:

  • üìç Batch Operations: Insert/update/delete multiple rows in one statement
  • üìç Use Indexes: Ensure WHERE clause columns are indexed
  • üìç Disable Keys: For bulk inserts, temporarily disable indexes
  • üìç Use LOAD DATA: For importing large datasets from files
  • üìç Limit Scope: Use LIMIT for large updates/deletes

The DML Safety Checklist

Before running any DML statement in production:
  1. ‚úì Test on a development database first
  2. ‚úì Run SELECT with same WHERE clause
  3. ‚úì Create a backup or use transactions
  4. ‚úì Double-check the WHERE clause
  5. ‚úì Consider the impact on related tables
  6. ‚úì Check for triggers that might fire
  7. ‚úì Verify you're connected to the right database
  8. ‚úì Have a rollback plan ready

Common Mistakes to Avoid

  • ‚ùå Forgetting WHERE clause in UPDATE/DELETE
  • ‚ùå Not testing complex conditions first
  • ‚ùå Ignoring foreign key constraints
  • ‚ùå Not checking affected row count
  • ‚ùå Running DML during peak hours
  • ‚ùå Not considering trigger side effects
  • ‚ùå Assuming NULL behaves like other values
  • ‚ùå Not backing up before bulk operations

The Four Essential SQL Statements

Statement Purpose Risk Level Can Undo?
SELECT Read data Safe N/A
INSERT Add new rows Medium DELETE the rows
UPDATE Modify existing rows High Only with backup

Key Concepts to Remember

INSERT Statement

  • Can insert single or multiple rows
  • Use DEFAULT keyword or omit columns for default values
  • NULL means "no value", different from empty string
  • Can use subqueries for complex inserts
  • Handle duplicates with IGNORE, ON DUPLICATE KEY, or REPLACE

UPDATE Statement

  • ALWAYS use WHERE unless updating all rows
  • Test with SELECT first
  • Can use calculations and functions in SET
  • Can update based on other tables using subqueries
  • Check ROW_COUNT() to verify changes

DELETE Statement

  • Most destructive operation - use with extreme caution
  • Consider soft deletes for important data
  • TRUNCATE is faster for removing all rows
  • Can use subqueries for complex conditions
  • Always backup before bulk deletes

Auto-Commit Reminder

Remember: MySQL auto-commits by default. Each DML statement is immediately permanent unless you explicitly use transactions. To work more safely:
  • Use START TRANSACTION for grouped changes
  • Consider SET autocommit = 0 for session safety
  • Learn about transaction isolation levels (Chapter 14)
