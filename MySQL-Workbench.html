Now we'll connect the tables. We'll create 18 relationships total.

IMPORTANT: When you draw relationship lines between tables, MySQL Workbench AUTOMATICALLY:

  • Creates the foreign key constraint
  • Adds "FK" designation to the column
  • Sets up the proper references
  • Creates indexes for performance

You don't need to manually set foreign keys - the relationship tools do it all for you!

Understanding Relationship Lines in MySQL Workbench:

MySQL Workbench has TWO types of relationship tools:

  • ‚îÅ‚îÅ‚îÅ Solid Line (Identifying Relationship) - Tool icon: solid line with crow's foot
    • Used when the foreign key becomes PART of the primary key
    • Child record cannot exist without parent
    • Example: Order_Items needs both order_id AND product_id
  • ‚îÖ‚îÖ‚îÖ Dashed Line (Non-Identifying Relationship) - Tool icon: dashed line with crow's foot
    • Used when the foreign key is just a regular column
    • Child record can conceptually exist without parent
    • Example: Products can exist without a Category

How to Create a Relationship:

  1. Select the correct tool from the left toolbar (solid or dashed as specified below)
  2. Click the PARENT table first (the "one" side - table with the primary key)
  3. Click the CHILD table second (the "many" side - table with the foreign key)
  4. MySQL Workbench automatically creates the foreign key column if it doesn't exist

Visual Guide to Tools:

In the MySQL Workbench toolbar, you'll see these relationship tools:

  • ‚îÖ‚îÖ‚îÖ‚î§ = 1:1 Non-Identifying (dashed line, one-to-one)
  • ‚îÅ‚îÅ‚îÅ< = 1:n Identifying (solid line, one-to-many)
  • ‚îÖ‚îÖ‚îÖ< = 1:n Non-Identifying (dashed line, one-to-many)

Troubleshooting Relationships:

  • If you can't find the right tool:
    • The tools are in the left vertical toolbar
    • Hover over each icon to see tooltips
    • Solid line = Identifying relationship
    • Dashed line = Non-Identifying relationship
    • Depending on the configuration of Workbench you have to click the child table first then the parent table to get the primary and foreign key correct.
  • If the relationship arrow is going the wrong way:
    • You clicked the tables in the wrong order
    • Delete the relationship (select it and press Delete key)
    • Try again, clicking the parent table first
  • If you see a dialog asking about columns:
    • This happens when column names don't match exactly
    • Select the correct columns from the dropdowns
    • For example: address_id ‚Üí ship_address_id
  • If the relationship won't create:
    • Check that both columns have the same data type (both INT)
    • Make sure the parent table has a primary key
    • Verify the column exists in both tables

üß† What's a Junction Table?

A junction table is basically a "relationship table" that connects two other tables when both sides can have many of each other.

It's like the middleman in a complicated relationship status.

üí¨ Think of it like:

"It's complicated ‚Äî we need a table for that."

üéµ Example: Playlists and Songs

You have:

  • A Playlists table
  • A Songs table

One playlist can have many songs, but one song can also appear in many playlists.

That's a many-to-many relationship.

You can't just add a single "playlist_id" inside the Songs table ‚Äî because each song could belong to lots of playlists.

So you make a junction table called, for example, Playlist_Songs.

playlist_id | song_id -------------|---------- 1 | 10 1 | 14 2 | 10 2 | 18

Now:

  • Playlist #1 has songs #10 and #14
  • Playlist #2 has songs #10 and #18
  • Song #10 is in both playlists

üß© How It Works

That junction table just has two foreign keys ‚Äî one from each main table:

  • One connects to Playlists.playlist_id
  • One connects to Songs.song_id

Together, they show the connection between the two.

üí° Why It's Important

  • It keeps your database organized and normalized
  • You don't repeat data all over the place
  • You can easily look up connections between things
  • You can add extra columns like "date_added" or "rating" if you want more info about the relationship itself

üîë TL;DR

A junction table:

  • Lives between two tables
  • Handles many-to-many connections
  • Has two foreign keys
  • Keeps everything clean, flexible, and easy to query

Basically ‚Äî it's the "matchmaker" of your database üíò

Understanding the Design

Why Junction Tables?

Junction tables (Order_Items, Shopping_Cart, Wishlists) solve many-to-many relationships:

  • Orders ‚Üî Products: An order contains many products; a product appears in many orders
  • Customers ‚Üî Products (Cart): Customers can cart many products; products can be in many carts
  • Customers ‚Üî Products (Wishlist): Customers can wish for many products; products can be wished by many
